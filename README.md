Задание 1. СУБД
Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Приведите ответ в свободной форме.  
Ответы:  
1.1  Реляционные базы данных (RDBMS) – такие как PostgreSQL, MySQL, Oracle Database или Microsoft SQL Server. Эти системы обеспечивают строгую структурированность данных, поддерживают транзакционную целостность и ACID-свойства (атомарность, согласованность, изолированность, долговечность). Они идеально подходят для финансового анализа и отчетов благодаря поддержке сложных запросов и операций JOIN.  
1.1* Для решения задачи хеширования можно использовать специализированные СУБД, такие как Redis или Memcached,которые предоставляют API для быстрого доступа к данным в оперативной памяти. Они могут значительно ускорить процесс хеширования и обеспечить быстрый доступ к результатам.  
1.2  Документно-ориентированные базы данных (NoSQL) – MongoDB, Couchbase или Amazon DynamoDB. Такие базы хорошо работают с полуструктурированными данными, позволяют легко добавлять новые поля и обеспечивают высокую производительность при чтении и записи больших объемов данных.  
     Для CRM также подойдет реляционная база данных, особенно если необходимы точные связи между сущностями (например, клиенты и заказы).  
1.2* Да можно установить для этой задачи одну базу данных, можно использовать реляционную базу данных типа PostgreSQL или MySQL, которые способны эффективно обрабатывать как финансовые данные, так и работу с полуструктурированными данными (через JSONB).  
     Другой вариант – гибридные базы данных, такие как CockroachDB, которые объединяют преимущества реляционных и NoSQL решений.  
1.3    Подходящие типы СУБД:  
     Файловые хранилища – SQLite, если объем данных небольшой и доступ осуществляется локально.  
     Реляционные базы данных, если ожидается значительный рост объема данных и необходимость централизованного доступа.  
1.3* Да, можно использовать MongoDB, если выбрали ее для лендингов и CRM. В этом случае корпоративные нормы и правила можно хранить в отдельных коллекциях в виде структурированных JSON-документов.  
1.4  Графовые базы данных – Neo4j, OrientDB или ArangoDB. Графовые базы отлично справляются с задачами маршрутизации и распределения ресурсов, поскольку оптимизированы для быстрого поиска связей между объектами.  
1.4* Можно подключить отдел закупок к СУБД логистов, особенно если используется графовая или пространственная СУБД. Данные о закупках (поставщики, материалы, цены, сроки) можно связать с данными о логистике (объекты, маршруты, курьеры) для оптимизации поставок. Важно правильно спроектировать схему данных и настроить API для обмена информацией.  
1.5* Теоретически возможно объединить все задачи в рамках одной СУБД, можно выбрать реляционную БД по типу PostgreSQL. Однако на практике рекомендуется использовать разные типы СУБД для разных задач, чтобы добиться оптимальной производительности и надежности.  
     Например:  
     -Реляционная база данных для финансовой аналитики и отчетности.  
     -Документно-ориентированная база для работы с CRM и сбора данных с лендингов.  
     -Графовая база для задач логистики и формирования маршрутов.  
     Это позволит обеспечить хорошую производительность и управление данными.
