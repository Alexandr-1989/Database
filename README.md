Задание 1. СУБД
Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Приведите ответ в свободной форме.  
Ответы:  
1.1  Реляционные базы данных (RDBMS) – такие как PostgreSQL, MySQL, Oracle Database или Microsoft SQL Server. Эти системы обеспечивают строгую структурированность данных, поддерживают транзакционную целостность и ACID-свойства (атомарность, согласованность, изолированность, долговечность). Они идеально подходят для финансового анализа и отчетов благодаря поддержке сложных запросов и операций JOIN.  
1.1* Для решения задачи хеширования можно использовать специализированные СУБД, такие как Redis или Memcached,которые предоставляют API для быстрого доступа к данным в оперативной памяти. Они могут значительно ускорить процесс хеширования и обеспечить быстрый доступ к результатам.  
1.2  Документно-ориентированные базы данных (NoSQL) – MongoDB, Couchbase или Amazon DynamoDB. Такие базы хорошо работают с полуструктурированными данными, позволяют легко добавлять новые поля и обеспечивают высокую производительность при чтении и записи больших объемов данных.  
     Для CRM также подойдет реляционная база данных, особенно если необходимы точные связи между сущностями (например, клиенты и заказы).  
1.2* Да можно установить для этой задачи одну базу данных, можно использовать реляционную базу данных типа PostgreSQL или MySQL, которые способны эффективно обрабатывать как финансовые данные, так и работу с полуструктурированными данными (через JSONB).  
     Другой вариант – гибридные базы данных, такие как CockroachDB, которые объединяют преимущества реляционных и NoSQL решений.  
1.3    Подходящие типы СУБД:  
     Файловые хранилища – SQLite, если объем данных небольшой и доступ осуществляется локально.  
     Реляционные базы данных, если ожидается значительный рост объема данных и необходимость централизованного доступа.  
1.3* Да, можно использовать MongoDB, если выбрали ее для лендингов и CRM. В этом случае корпоративные нормы и правила можно хранить в отдельных коллекциях в виде структурированных JSON-документов.  
1.4  Графовые базы данных – Neo4j, OrientDB или ArangoDB. Графовые базы отлично справляются с задачами маршрутизации и распределения ресурсов, поскольку оптимизированы для быстрого поиска связей между объектами.  
1.4* Можно подключить отдел закупок к СУБД логистов, особенно если используется графовая или пространственная СУБД. Данные о закупках (поставщики, материалы, цены, сроки) можно связать с данными о логистике (объекты, маршруты, курьеры) для оптимизации поставок. Важно правильно спроектировать схему данных и настроить API для обмена информацией.  
1.5* Теоретически возможно объединить все задачи в рамках одной СУБД, можно выбрать реляционную БД по типу PostgreSQL. Однако на практике рекомендуется использовать разные типы СУБД для разных задач, чтобы добиться оптимальной производительности и надежности.  
     Например:  
     -Реляционная база данных для финансовой аналитики и отчетности.  
     -Документно-ориентированная база для работы с CRM и сбора данных с лендингов.  
     -Графовая база для задач логистики и формирования маршрутов.  
     Это позволит обеспечить хорошую производительность и управление данными.  
       
Задание 2. Транзакции  
2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.  
2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?  

Приведите ответ в свободной форме.  
Ответ:  
2.1. Этот процесс состоит из шести основных шагов:  
1 Инициация транзакции: Пользователь инициирует операцию пополнения баланса, вводит номер телефона и сумму пополнения.  
2 Проверка введенных данных: Система проверяет правильность номера телефона и запрашиваемую сумму пополнения.  
3 Запрос на списание средств: Система отправляет запрос платежному сервису (банку, электронной платёжной системе) на списание указанной суммы с банковского счёта или электронного кошелька пользователя.  
4 Обработка платежа: Платежный сервис обрабатывает запрос и подтверждает успешное списание средств.  
5 Пополнение баланса: После подтверждения успешного списания система обновляет баланс телефонного счёта пользователя на указанную сумму.  
6 Отправка уведомления: Пользователю отправляется уведомление о завершении транзакции (SMS, email или push-уведомление).  
2.1.* Автоплатеж
Процесс пополнения счета через автоплатеж отличается автоматизмом и отсутствием прямого участия пользователя:  
1 Настройка автоплатежа: Пользователь настраивает автоплатеж, выбирая частоту пополнения (например, ежедневно, еженедельно, ежемесячно) и минимальный остаток на счету, при достижении которого должно происходить пополнение.  
2 Мониторинг остатка: Система периодически проверяет состояние счёта абонента и сравнивает его с установленным минимальным порогом.  
3 Автоматическое инициирование транзакции: Когда сумма на счете достигает установленного порога, система автоматически запускает процедуру пополнения.  
4 Проведение проверки и обработка платежа: Выполняются аналогичные шаги, как и при ручном пополнении: проверка правильности данных, отправка запроса на списание средств и подтверждение оплаты.  
5 Активация пополнения: Баланс счета автоматически увеличивается на заданную сумму.  
6 Уведомление пользователя: Как и при ручной оплате, пользователю отправляется уведомление о выполнении автоплатежа.  
Таким образом мы видим что процессы по шагам схожи, отличие идет в автоматизации и необходимостью вмешательства пользователя.  
  
Задание 3. SQL vs NoSQL  
3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.  
3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.  

Приведите ответ в свободной форме.  
Ответ:  
3.1:  
1 Строгость структуры данных:SQL-системы используют заранее определенную схему данных, что помогает поддерживать целостность и упорядоченность данных. Это упрощает создание сложных запросов и поддержание целостности данных внутри приложений.  
2 ACID-соответствие:Транзакции в SQL-системах соответствуют принципам атомарности, согласованности, изоляции и долговечности (ACID). Это гарантирует, что изменения данных будут выполнены либо целиком, либо не выполнятся вовсе, обеспечивая надёжность и устойчивость данных.  
3 Стандартизация и поддержка стандартов:Язык SQL является стандартизированным языком запросов, что облегчает переносимость приложений между различными системами и платформами. Многие разработчики знакомы с SQL, что снижает затраты на обучение персонала.    
4 Сложные аналитические возможности:SQL поддерживает мощные инструменты для выполнения сложных аналитических запросов и агрегации данных. Запросы SQL позволяют проводить глубокие анализы и создавать детализированные отчёты.  
5 Удобство поддержки и администрирования:Большинство популярных SQL-систем обладают развитым инструментарием для мониторинга, резервного копирования и восстановления данных. Это значительно упрощает процессы обслуживания и сопровождения базы данных.  
3.1*:  
NewSQL представляет собой новый класс баз данных, сочетающих лучшие черты традиционных SQL-систем и современные подходы горизонтального масштабирования, характерные для NoSQL-решений.  
Преимущества:  
1 Масштабируемость и отказоустойчивость:NewSQL-системы предлагают масштабируемость уровня NoSQL, позволяя горизонтальное расширение без потери доступности данных даже при значительных нагрузках.
2 Соответствие требованиям ACID:Несмотря на поддержку масштабирования, NewSQL сохраняет гарантии ACID-транзакций, обеспечивая целостные и надежные операции с данными.
3 Совместимость с SQL-запросами:Новый класс баз данных поддерживает стандартный SQL-интерфейс, облегчая переход от классических реляционных баз данных и снижая риски миграции существующих приложений.
4 Высокая производительность:Современные NewSQL-решения часто показывают превосходящую производительность благодаря оптимизации индексов, кэшированию и асинхронным операциям ввода-вывода.
5 Простота эксплуатации:Поскольку NewSQL объединяет характеристики SQL и NoSQL, управление этими системами становится проще: отсутствует необходимость выбирать между производительностью и надежностью, что уменьшает нагрузку на разработчиков и администраторов баз данных.  
  
Задание 4. Кластеры  
Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.  
На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?  

Приведите ответ в свободной форме.  
Ответ:  

Критерии выбора СУБД:  
1 Производительность и масштабируемость:нужно выбрать систему, способную эффективно распределять нагрузки между тысячами узлов, обеспечивать параллельную обработку данных и быстрое выполнение операций.  
2 Надёжность и доступность:база данных должна быть устойчивой к сбоям отдельных узлов, сохранять работоспособность даже при отказе части инфраструктуры.  
3 Поддержка репликации и шардинга:репликация обеспечит избыточность данных и повысит доступность, тогда как шардинг поможет равномерно распределить нагрузку и увеличить производительность.  
4 Тип хранимых данных: важно учесть природу данных — являются ли они структурированными, полуструктурированными или совсем неструктурированными. От этого зависит, подойдёт ли реляционная, документо-ориентированная или графовая база данных.  
5 Стоимость владения и простота эксплуатации:необходимо оценить стоимость лицензий, обслуживание, наличие квалифицированного персонала и уровень сложности настройки и эксплуатации выбранной системы.  
6 Интеграция с существующими приложениями:Убедитесь, что выбранная СУБД совместима с используемыми инструментами разработки и инфраструктурами предприятия.  
Учитывая количество машин и объём данных, оптимальным выбором станет одна из моделей распределителенных вычислений, ориентированных на массовую параллельную обработку больших массивов данных:  
  
1 Hadoop + HDFS (Apache Hadoop Ecosystem):Подходит для обработки больших объемов данных в пакетном режиме. Эта экосистема включает MapReduce для параллельной обработки данных и HDFS для эффективного хранения данных на кластере.  
2 Spark (Apache Spark):Отличается высоким уровнем производительности и поддержкой интерактивных запросов. Spark может использоваться поверх HDFS или других файловых систем, обеспечивая эффективную обработку данных как в пакетном, так и в потоковом режимах.  
3 BigTable-подобные системы (Google Bigtable, Apache Cassandra, HBase):Идеальны для высоконагруженных онлайн-сервисов, обеспечивающих быстрый доступ к большим наборам данных с использованием горизонтальной масштабируемости и распределённого хранения.  
4 Распределённые NoSQL-хранилища (MongoDB Sharding, ScyllaDB):Обеспечивают высокую пропускную способность и низкую задержку при обработке больших объёмов данных, предлагая эффективное хранение и обработку полуструктурированной информации.
